## 3. 故事深度分析 — 主题与模式

### 3.1 共性难点（多条故事反复出现）

1. **棕地 workspace 与可复现构建**  
   - 1.2：干净 clone 下全量测试曾因缺少默认 `skills/weather` 布局失败；通过补齐内置技能布局与测试假定对齐解决。  
   - **系统启示：** 故事完成定义应显式包含「最小可 clone 通过的测试矩阵」或与 CI 对齐的夹具说明。

2. **同一 diff / 分支上故事边界模糊**  
   - 1.1、1.2、1.3 的 Review 均提到：合并批次中混入后续故事改动（README、`lib.rs`、CHANGELOG），增加审查与 bisect 成本。  
   - **系统启示：** 尽量按故事切提交或 PR；File List 与「本故事仅交付 X」叙述冲突时，在工件中加注「后续已合入」避免审计混淆。

3. **契约与实现漂移**  
   - 1.3：`get_neuro_overview_snapshot` 与 v0 契约表不一致，经裁定后文档补全白名单与 JSON 示例。  
   - **系统启示：** 新增 command/DTO 时同步更新「契约单一文档」，避免 GUI/集成方按旧表集成。

4. **并发与测试环境**  
   - 1.3：`AGENT_DIVA_TEST_CORTEX_SYNC_FAIL` 并行竞态 → `serial_test` + `#[serial]`。  
   - 1.5：Tokio 默认单线程 runtime 与 MCP `block_in_place` 冲突 → 相关用例改用 `multi_thread`。  
   - **系统启示：** 环境变量与全局单例的测试须默认串行或隔离。

### 3.2 评审反馈中的重复主题

- **安全与 NFR：** 1.5 中 `tool_call_started` 的 message 需与 `tool_call_finished` 一致走 `sanitize_tool_summary_for_process_event`（NFR-S2 精神）。  
- **可观测性：** `cortex_toggled` 发射失败从静默改为 `tracing::warn!`。  
- **契约一致性：** 1.8 神经总览 `detail` 中 `stop_reason` 须与 JSON camelCase wire 字面量一致，而非 `Debug`。  
- **算法顺序：** 1.8 收敛循环「先 `is_done` 再预算触顶」修复 `max_internal_rounds == 1` 误报 `BudgetExceeded` —— 说明边界条件需专用回归测试。

### 3.3 显式延后项（技术债登记，供 Epic 2+）

| 来源故事 | 延后项 | 备注 |
|----------|--------|------|
| 1.1 | README 指向 `agent-diva-swarm/docs` 在仅克隆子仓时可能 404 | 文档注明 monorepo 布局 |
| 1.4 | `full_swarm_cap_observable_via_pipeline_without_gui` 未进 1.4 测试对照表 / 无 `doc-ref` | 与 1.8 范围重叠，建议在整合文档补一行 |
| 1.5 | `ProcessEventBatchSink::deliver_batch` 若同步阻塞则拉长 turn | trait/文档约定下游轻量 |
| 1.6 | `ValidatedManifest` 直接 `Deserialize` 可绕过校验；registry 锁中毒时查询静默 | v0 可接受；生产化时收紧 |
| 1.7 | Light 路径上限在最小 turn 桩未强制墙钟/步数 | 注释已说明「后续接入运行时」 |
| 1.8 | `Timeout` 在收敛循环中预留但未接线 | stub 范围；接线后补测 |

### 3.4 突破与可复用做法

- **单文件轻量意图规则**（`light_intent_rules.rs`）+ `docs/adr-e-fr19-execution-tier.md`：满足 FR19「单点维护、可测」。  
- **过程事件管道**：白名单 DTO + 皮层门控 + 节流，对齐 NFR-I2 / NFR-P2。  
- **FR21 与皮层 OFF 合并语义**（`ADR_FR21_FORCE_LIGHT_AND_CORTEX_OFF.md`）+ 具名无头测试：消除 FR3/FR21 长期歧义。  
- **简化模式文档 + `doc-ref` + 故意走错分支用例**（1.4）：把「可 grep 开/关」写进验收 DNA。

## 6. 史诗回顾讨论摘要

### 6.1 做得好的地方

- **架构边界：** `agent-diva-swarm` 保持 ADR-A（不依赖 `agent-diva-meta`），与规划一致。  
- **FR12 / 无头优先：** 1.4、1.7、1.8、1.9 均以 `cargo test` 可重复验收关键分支。  
- **评审闭环：** 多轮 bmad-code-review 将真实 bug（收敛顺序、序列化 wire、sanitize）挡在合并前。  
- **文档交叉引用：** `CORTEX_OFF`、FR21 ADR、IR 报告「实现冻结」行形成可追溯链。

### 6.2 挑战与改进方向（对事不对人）

- **批次过大：** 减少「一个 PR 含多故事 + 无关签名变更」；利于回滚与责任清晰。  
- **契约与代码同步：** 新 command 默认进入契约表的「门禁」可写入 DoD 或 CI 检查清单。  
- **测试矩阵与仓库布局：** 文档化「完整 monorepo vs 子仓 clone」的差异，降低新贡献者摩擦。

### 6.3 关键洞察

1. **在棕地上，「能编译」不等于「能 clone 即测」** —— 技能目录等隐式假定要显式化或夹具化。  
2. **过程事件与安全/性能同源设计** —— sanitize、节流、flush 策略应和事件类型一起在评审中过一遍。  
3. **FR19–FR21 分拆正确** —— 路由（1.7）、收敛（1.8）、强制轻量冻结（1.9）分故事降低了并行误解。
