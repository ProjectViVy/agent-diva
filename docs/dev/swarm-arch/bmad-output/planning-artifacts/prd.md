# Product Requirements Document - newspace

**Author:** Com01
**Date:** 2026-03-30

## Executive Summary

本 PRD 描述在既有 **agent-diva**（多 Rust crate + **agent-diva-gui** / Tauri）上的棕地增量：**Person 中心**的蜂群编排与观测能力。对用户叙事为 **单一「人」**（单一责任与对外口径）；**对内** 保留多参与者协作（handoff、黑板、合成等），协作单元以 **能力（Capability）** 装备与注册为主，**性格/心态** 后置。交付形态以 **开发者向运行时（crate）** 为主，**首要用户触点** 为桌面 GUI；GUI 中的 **神经系统** 视图承载对内连接与信号/拓扑的可视化，**界面形态分前期（架构图式大脑、尽量简单）与后期（总控台 + 游戏化愿景）**，详见 **「神经系统 UI 分期设计」**；**中控台** 由产品扩展，且不占用神经系统占位区。参考 OpenAI Agents、Shannon、OMC 等的设计密度，但 **运行与交付不依赖外部 IDE 宿主**，能力全量 **内化在 agent-diva**。v0 强调 **先跑通、控范围**：默认合并与最简协议优先，复杂租约、全量安全与 hooks 深化跟随主线迭代。

**关键体验：**（1）在用户与系统 **交互过程中**，GUI 应能呈现 **过程本身**（编排与对内协作的可视反馈，而非仅事后日志）；粒度与刷新策略在后续范围与 NFR 中定义。（2）在 **聊天主页** 提供 **「大脑皮层」** 开关（**大脑图标** 标记），用于 **打开或关闭蜂群功能**；关闭时的运行时语义（例如是否退化为严格单路径对话、对内协作是否完全暂停）须在 **功能范围** 中单点定义并验收。

### What Makes This Special

- **语义重定向**：对外「一个人」，对内仍为蜂群；避免多聊天机器人头像式体验，同时不阉割协作。
- **能力优先**：能力为可声明、可挂载的单元（初期可贴近 skills，可演进为更大能力包）；用户可自定义能力包；Voice 可自定义；Chair 用户规则为后期趣味能力。
- **桌面可感知 + 可控**：神经系统与（扩展中的）中控台支撑观测；**大脑皮层** 开关让用户在 **主聊天界面** 即可 **显式启用/停用** 蜂群层，平衡「要深度」与「要简单」。
- **路径与成本纪律**：**简单意图**（如单技能直调、短问答）默认 **不** 被拖入完整多参与者蜂群；蜂群路径须有 **收敛与完成定义**，避免无上限「思考—博弈」消耗调用与配额（见 User Journeys 旅程五～六、FR19–FR22、NFR-P3）。
- **诚实工程边界**：核心状态与编排语义在 **Rust**；GUI 为 **稳定 API 的消费者**；集成面受控以防「通用」域无限膨胀。

## Success Criteria

### User Success

- **可感知：** 在 **「大脑皮层」开启** 且用户与 DiVA 交互时，GUI 能在任务过程中呈现 **编排/对内协作的过程反馈**（非仅最终回复），具体形态（时间线、阶段条、事件流等）在实现阶段选定，但验收上须明显优于「只有聊天记录」。
- **可控：** 用户在 **聊天主页** 通过 **大脑图标开关** 明确感知当前是否启用蜂群层；切换后行为与文案/提示一致，无静默失效。
- **可诊断：** 用户能进入 **「神经系统」** 相关界面，理解当前连接/活动或异常占位信息；v0 至少 **可导航 + 与真实或 stub 数据绑定其一**。
- **可扩展：** 进阶用户能继续通过既有或新增路径 **管理 skills / 能力包**（与「能力优先」路线一致），且不破坏「关大脑皮层」时的简化路径。
- **不空转：** 在 **轻量意图** 下，用户能在 **合理时间与步数内** 得到 **结果或明确失败原因**，而非长时间内部推理无产出（与 FR19–FR20 对齐）。

### Business Success

- **叙事一致：** 对外强化 **「一个人 + 可选蜂群层」**，减少多 agent 产品常见的信任与解释成本。
- **支持效率：** 通过观测与开关，降低因「不知道系统在干什么」导致的排障与弃用（定性目标，可在发布后收集反馈验证）。

### Technical Success

- **架构边界：** 新增蜂群相关 **Rust crate** 与现有 agent-diva crate **单向依赖**；默认可通过 `cargo` 构建。
- **可测性：** 核心编排语义（含「大脑皮层」开/关分支）具备 **不依赖 GUI** 的自动化验证路径（单元或集成测试）。
- **契约稳定：** GUI 仅通过 **稳定 API / 事件** 消费状态；开关与后端状态 **单一真相源**，避免双端漂移。

### Measurable Outcomes

| 结果 | 验收要点 |
|------|----------|
| MVP 可用 | 聊天主路径可用；**大脑皮层** 开关可用且 **开/关** 行为在 PRD 范围节已定义并通过用例验收 |
| 过程可视 | 至少 **1** 类进行中反馈组件与后端事件或状态对齐（可 stub，须约定替换路径） |
| 神经系统 | 侧栏入口可达；界面 **非**「永久即将推出」空壳，或明确标注数据阶段并有关闭路径 |
| 回归 | 现有 gateway/聊天核心流程 **无新增 P0** |

## Product Scope

### MVP - Minimum Viable Product

- **大脑皮层** 开关 UI（聊天主页、大脑图标）及 **开/关运行时语义** 的明确定义与测试。
- **过程可视** 的最小实现（与现有流式/工具事件或新事件总线挂钩，择一落地）。
- **神经系统** 视图：**替换** `NormalMode` 中原「即将推出」占位；**首屏** 为 **大脑架构图式** 主视图（**左右分区**），见 **「神经系统 UI 分期设计」**；提供 **最小可用** 展示或分阶段标注。
- **新蜂群 crate** 骨架接入 workspace，承担与开关、过程事件相关的 **真相源或适配层**（具体边界与现有 `agent-diva-agent`/gateway 对齐实现）。
- **文档：** 本 PRD 范围节与 ADR/实现说明同步「关大脑皮层」行为。

### Growth Features (Post-MVP)

- 神经系统 **完整拓扑**、近实时更新与异常高亮。
- **中控台** 模块化扩展（能力、心态等，由产品路线驱动）。
- **Voice** 自定义、**Chair** 用户可配置规则（趣味/高级功能）。
- 更丰富的 **能力 manifest**（优先级、冲突解析、依赖）与 **doctor** 诊断联动。

### Vision (Future)

- **性格 / 心态** 层与调度、预算联动。
- **Meta / hooks** 内化对标 OMC 级覆盖（事件全生命周期）。
- 多进程/分布式蜂群（若与产品叙事仍兼容 Person 对外模型）。

### Version roadmap & 对标边界（1.0.0 发布清单 — 2026-03-31 冻结）

**Why：** 避免「自研叙事」在 **1.0.0** 前稀释验收标准；对外可解释 **我们先把行业已验证的多代理编排基线做扎实**，再叠 IDE/工作流元层，最后才做 **下一代自有系统**。

**分阶段立场（产品顺序，后者不得挤占前者必达项）：**

| 阶段 | 目标 | 与当前 PRD 关系 |
|------|------|------------------|
| **→ 1.0.0** | 交付物须 **达到可对外声明的「多代理编排基线」**：同时满足 **传统 Swarm-类** 与 **Shannon-类** 在本节下列出的 **P0 必达项**（在 `architecture.md` 或独立 **1.0.0 发布清单** 中维护为可勾选表格；本 PRD 只冻结 **原则**）。 | 现有 FR19–FR22、大脑皮层、过程事件、Epic 5 阶段 B 等 **服务于** 本条；**未勾满 P0 不得标 1.0.0**。 |
| **1.0.0 之后 → OMC 向** | 在 **不破坏** Person 叙事与 Rust 真相源前提下，**工作流 / hooks / 插件式体验** 向 **oh-my-claude-code（OMC）** 级对齐（事件生命周期、技能与魔法词等 —— **详细范围另起 Growth 条目**，**不进入 1.0.0 必达**）。 | 对应原 Vision 中「Meta / hooks 内化」的 **显式阶段化**；避免与 1.0.0 抢资源。 |
| **再之后 — 下一代自有栈** | **自研编排、自有协议或差异化 UX** 仅在 **前两阶段已交付** 后作为 **Vision+** 立项；不在 1.0.0 讨论范围内兜底承诺。 | 创新章节可演进，**不得反向削弱 1.0.0 清单**。 |

**1.0.0 P0 必达项 — 双轨对照（摘要；细则以发布清单附件为准）：**

- **Swarm-类（handoff / 多角色编排）P0：** 皮层 ON 时存在 **非「仅 spawn 子代理」** 的可观测多步编排路径；**显式** 轻量 vs 全量分层（FR19）；**handoff/序曲** 可配置或可扩展（与 Epic 5 一致）；对外仍 **单一 Person 叙事**（FR8/FR9）。
- **Shannon-类（生产向纪律）P0：** **有界编排**（轮次/预算/收敛，FR20、NFR-P3）；**失败与触顶** 可分类、可观测；**能力/工具边界** 可声明与诊断挂钩（与 FR10–FR11、FR18 精神一致）；**无 GUI** 可测核心分支（FR12）。

**说明：** 「双轨」不是二选一偷懒 —— **1.0.0 = 两套 P0 均通过**。若某条 P0 在实现中发现冲突，走 **Correct Course** 修订清单，**不得静默降标**。

## User Journeys

### 旅程一：进阶用户 — 打开「大脑皮层」跑复杂任务（成功路径）

**开场：** 用户在写代码或排障，希望 DiVA 能多步推理并可能触发工具链。  
**发展：** 在聊天主页打开 **大脑图标（大脑皮层 ON）**，发送任务；进行中出现 **过程反馈**（阶段或事件），用户能判断系统仍在推进或卡点位置。  
**高潮：** 任务完成，结果符合预期，且用户感到比「纯黑盒聊天」更可信赖。  
**结局：** 用户在对重任务时倾向保持大脑皮层开启。

### 旅程二：同一用户 — 要快或要简（大脑皮层 OFF）

**开场：** 用户希望低延迟或不需要内部过程信息。  
**发展：** 在聊天主页 **关闭大脑皮层**；对话遵循 PRD 已定义的 **简化路径**（具体语义以实现为准）。  
**高潮：** 响应符合「简单模式」预期，未意外触发蜂群层行为。  
**结局：** 用户理解何时应关闭开关。

### 旅程三：进阶用户 — 「神经系统」排障

**开场：** 任务卡住或结果异常。  
**发展：** 从侧栏进入 **神经系统** 视图；首屏为 **大脑架构图式** 左右分区，再查看连接、活动或异常信息（v0 可为最小数据集）。  
**高潮：** 用户获得可行动线索（例如某阶段未推进、工具未返回）。  
**结局：** 用户选择重试、调整提示或切换大脑皮层模式后再试。

### 旅程四：集成/开发者 — 无 GUI 也可信

**开场：** CI、自动化或 headless 场景。  
**发展：** 通过 crate、API 或测试验证 **大脑皮层开/关** 与核心编排语义。  
**高潮：** 自动化测试通过，发布前有信心。  
**结局：** 「GUI 非唯一真相源」成立。

### 旅程五：挫败路径 — 「只想调个技能 / 问句简单的」，蜂群却空转

**开场：** 用户意图明确且轻（例如 **调用某一 skill、一句可答问题**），期望 **短路径** 出结果。  
**发展：** 系统在 **大脑皮层开启** 或默认路由下仍进入 **重度多参与者编排**，长时间 **内部思考、博弈或互相等待**，用户侧 **看不到有效进展或始终无终态**。  
**高潮（负面）：** 用户认为产品「小题大做」、不可预测，转而关闭应用或强制关大脑皮层。  
**结局（产品须避免）：** 信任下降；与 **FR19、FR20、FR21** 及 **大脑皮层 OFF** 语义共同约束 —— **轻量意图不得默认全量蜂群**。

### 旅程六：挫败路径 — 高频使用蜂群导致 token / 成本不可接受

**开场：** 用户 **每天** 依赖 DiVA 处理大量任务，其中 **多数本可轻量完成**。  
**发展：** 每次请求都附带 **不必要的对内多轮推理、审查或对弈**，账单或配额 **快速耗尽**；用户感知 **边际收益远低于边际成本**。  
**高潮（负面）：** 用户降级为竞品或 **永久关闭蜂群**，即使复杂任务上蜂群仍有价值。  
**结局（产品须避免）：** 需在 **默认策略 + 可配置预算 + 可观测挂点** 上收敛（**FR20–FR22、NFR-P3**）；**Growth** 可扩展完整成本面板，**MVP** 至少 **不依赖无上限内部对话** 作为唯一完成手段。

### Journey Requirements Summary

| 区域 | 由旅程推出的需求 |
|------|------------------|
| 聊天主页 | 大脑皮层开关、当前模式可感知 |
| 过程可视 | 进行中反馈、与后端事件/状态对齐 |
| 神经系统 | 导航、**MVP 首屏大脑架构图 + 左右分区**、最小数据、异常/空闲态 |
| 运行时 | ON/OFF 分支、可测、单一状态源 |
| 意图与收敛 | **轻量路径默认**、蜂群 **步数/轮次上限**、**完成定义**、可选 **仅轻量路径**（FR19–FR21） |
| 成本与可观测 | **预算/警告挂点**（FR22、NFR-P3）；完整计费 UI 可 Post-MVP |
| 验收 | 每类旅程至少一条可自动化或手动用例（v0 可部分依赖 stub）；旅程五～六 各至少一条 **负向用例** 或 **阈值测试** |

## Domain-Specific Requirements

本 PRD 分类为 **general** 领域，**不适用**医疗、金融、政务等强监管专章。复杂度为 **medium** 主要来自 **多模块编排与桌面应用交付**，而非行业合规。

- **安全与隐私：** API Key、本地配置、日志与遥测遵循 **agent-diva 仓库主线** 及目标平台商店/分发要求；细则不在本 PRD 逐条立法。  
- **集成：** LLM 供应商、MCP、skills 等沿用既有配置模型；新增蜂群事件/开关须 **不破坏** 现有配置契约。  
- **风险缓解：** 「通用」域易导致范围膨胀 —— 以 **MVP 白名单**、集成面受控与成功标准中的可测结果约束（见上文）。

## Innovation & Novel Patterns

### Detected Innovation Areas

- **语义重定向：** 单进程、单对外叙事下保留 **对内蜂群协作**（能力装备、合成），产品话术统一为「一个人」，区别于多聊天机器人式 swarm。
- **大脑皮层开关：** 在 **主聊天界面** 用 **大脑图标** 显式启用/停用蜂群层，使用户在同一应用内切换「深度编排」与「简化路径」。
- **神经系统观测面：** 将 **对内拓扑与过程** 从纯日志提升到 **专用 GUI 视图**，并与 **进行中过程可视** 联动。
- **内化对标：** 吸收 OMC / Shannon / Agents SDK 等思路，**交付不绑定外部 IDE 宿主**，能力在 agent-diva 内一体化。

### Market Context & Competitive Landscape

- 多 agent 方案偏 **集成工具链** 或 **多角色对话展示**；本方案强调 **Person 外壳 + 可选蜂群内核 + 桌面可控**，差异化在 **叙事与显式开关**。

### Validation Approach

- **技术：** 「大脑皮层」开/关分支的自动化测试；过程可视与事件源对齐的集成测试。
- **产品：** 目标用户能否在短时间内理解开关含义并完成 ON/OFF 各一条成功路径（走查或轻量访谈）。
- **降级预案：** 过程可视若噪声过大，可退化为 **关键阶段节点** 而非全量细粒度事件。

### Risk Mitigation

- **复杂度过高：** MVP 白名单 + 「关大脑皮层」简化路径兜底。
- **概念过载：** 神经系统与中控台分坑；附录集中术语。
- **实现漂移：** 核心语义以 **Rust 真相源** 为准，GUI 仅消费稳定 API。

## 神经系统 UI 分期设计

> **来源：** 产品讨论（PM 结构化）；与上文 **MVP / Growth / Vision** 及附录 **神经系统 vs 中控台** 一致。同一套 **连接/活动/拓扑** 信息模型可在两阶段用不同 **深度与隐喻** 呈现；**MVP 仅验收前期**。

### 分期总览

| 阶段 | 入口后体验（目标） | 与产品范围关系 |
|------|-------------------|----------------|
| **前期（MVP）** | 用户从主导航进入神经系统后，**直接**看到 **DIVA 大脑** 主视图：**左右分区**（左脑/右脑或产品定义下的等价命名），整体为 **简单架构图风格**（区块、连线、标签清晰可读），**非** 游戏 HUD、**非** 多角色动画场景。 | 与 **MVP** 中「神经系统非空壳、最小可用展示」对齐 |
| **后期（Growth / Vision）** | 用户进入后 **先进入总控台**（hub），整体布局 **偏游戏化**；愿景为《头脑特工队》式 **强大、好玩** 的 **控制中心**，多「角色」/小人 **忙碌感** 与 **真实运行时状态** 对齐（避免纯装饰）。 | 与 **Growth**（中控台模块化等）及 **Vision** 联动；**不** 作为 MVP 交付条件 |

### 前期（尽量简单）— 交互与视觉边界

- **首屏：** 单一落地页级 **大脑示意图 + 左右分区**；用户 **无需** 先经过总控台或关卡式地图。
- **导航：** 点击 **分区** 可展开 **列表或侧栏**（模块、能力、连接或状态条目—具体字段与 API 对齐实现），满足 **可诊断**（与 FR6、FR7 一致）。
- **允许：** 轻量动效（如 **高亮、淡入、连线强调**）；静态或准静态 **架构图** 美学优先。
- **明确不做（MVP）：** 总控台式 hub 首屏、小人/角色动画、粒子或「繁忙场景」叙事、多面板游戏 UI；上述列入 **后期**。

### 后期（愿景）— 信息架构与体验说明（非 MVP）

- **入口 IA：** **总控台** 作为 **第一层**；从总控台再进入 **大脑** 或其他子视图（具体导航树由 UX/实现迭代，本 PRD 只锁定 **与 MVP 的差异**：后期 **不是** 「直达大脑」单屏）。
- **体验目标：** **游戏感布局** + **拟人化忙碌** 须 **映射** 真实任务/队列/协作状态（例如活动、等待、错误），以便 **可信 + 好玩**；细节在 **Growth** 阶段 PRD 增量或 UX 规格中展开。

### 待澄清（实现前建议产品/UX 收敛）

1. **前期首屏** 用户主目标是 **理解拓扑结构**、**操作/调试**，还是二者兼有？若偏操作，须在 MVP 内为关键动作保留 **最小控件**（不改变「架构图优先」原则）。
2. **左脑 / 右脑** 的 **业务语义** 须在 GUI 文案与帮助中写清，避免被理解成字面神经科学分区。
3. **后期** 总控台与「大脑」视图是 **父子（hub → 子视图）** 还是 **并列入口**，影响导航与深层链接策略，在启动 Growth 前决议。

## Functional Requirements

> 能力合同：未列能力默认 **不在** 本期交付范围；实现方式可多解，验收以可测行为为准。

### 蜂群模式与对话

- **FR1：** 用户可以在 **聊天主页** 通过 **大脑皮层** 控件 **启用或停用** 蜂群层。
- **FR2：** 系统在 **大脑皮层启用** 时，能够在用户任务 **进行过程中** 在 GUI 提供 **非仅最终文本** 的过程反馈（阶段/事件类，最小一种即可）。
- **FR3：** 系统在 **大脑皮层停用** 时，行为符合 **本文档或链接的实现说明** 中定义的 **简化模式**（须可测）。
- **FR4：** 用户可以在 **不切换应用** 的情况下，感知当前处于 **大脑皮层开或关** 状态。

### 神经系统与观测

- **FR5：** 用户可以从主导航进入 **神经系统** 视图（替换原「即将推出」占位）。
- **FR6：** 神经系统视图能够展示 **与当前会话或运行时相关的** 连接/活动或 **占位+说明**（v0 允许部分 stub，须标明数据阶段）。
- **FR7：** 用户可以在神经系统或关联界面获得 **排障线索**（例如空闲、错误、未推进），与聊天记录互补。
- **FR15：** 在 **MVP** 内，用户从主导航进入 **神经系统** 后，首屏须为 **DIVA 大脑** 的 **架构图式** 主视图，并包含 **左右两个可区分区域**（命名与语义见实现文案，须 **非空标签**）；**不得** 将「游戏化总控台」或「多角色忙碌场景」作为 MVP 必经首屏。
- **FR16：** **游戏化总控台优先入口**、**《头脑特工队》式多角色忙碌** 等体验 **不属于** MVP 验收范围；若实现中做占位，须 **明确标注为愿景/后续** 且 **不阻断** FR5–FR7、FR15 的 MVP 路径。

### Person 与对外叙事

- **FR8：** 系统在用户可见渠道上维持 **单一 Person 叙事线**（不出现多个并列「机器人头像」式独立对话流）。
- **FR9：** 系统在 **大脑皮层启用** 时仍满足 FR8，对内协作 **不** 以多用户可见聊天室形式暴露。

### 能力与配置（v0 边界内）

- **FR10：** 用户可以通过 **既有或随 MVP 提供的入口** 管理 **skills 或能力包**（与 agent-diva 当前设置能力衔接；具体入口以实现为准）。
- **FR11：** 系统能够 **加载并校验** 用户提供的 **能力声明**（v0 字段子集，错误时可见反馈）。

### 运行时与集成

- **FR12：** 系统能够在 **无 GUI** 场景下（测试或 headless）验证 **大脑皮层开/关** 与核心编排分支。
- **FR13：** GUI 仅通过 **已文档化的 API/事件** 获取蜂群状态与过程数据，**不** 将编排核心逻辑仅实现于前端。
- **FR14：** 系统能够将 **大脑皮层状态** 与 **gateway/后端** 状态保持 **单一真相源**（无长期双端不一致）。

### 文档与诊断

- **FR17：** 维护者可以查阅 **与 MVP 对齐** 的说明，解释 **关大脑皮层** 的语义与边界。
- **FR18：** 系统或工具链支持 **基本的自检/诊断输出**（随 diva 主线 `doctor` 或等价物演进，本 PRD 要求 **存在可扩展挂点** 而非一次做全）。

### 执行路径、收敛与成本纪律

- **FR19：** 当用户意图落在 **轻量类**（包括但不限于：**显式 skill 调用**、**短问答**、或文档化清单中的等价场景）时，系统 **不得** 在无 **显式用户选择** 的情况下启动 **完整多参与者蜂群编排**；须走 **可完成** 的轻量路径，并在 **文档化的超时或内部步数上限** 内 **交付结果** 或 **显式失败原因**（判定规则须在实现说明中单点维护，供测试引用）。
- **FR20：** 在 **大脑皮层启用** 且走蜂群编排时，系统须实施 **内置收敛策略**（至少包含：**最大内部轮次或等价预算**、**完成定义（done）**），**禁止** 依赖无终止条件的「思考—推翻—再思考」循环作为默认行为；默认阈值可在配置或常量中定义，并写入维护者文档。
- **FR21：** 用户或配置须能 **显式强制轻量路径**（可与「大脑皮层 OFF」合并为同一语义，或作为独立策略 —— 实现二选一须在链接文档中冻结），使得在强制期间 **不** 为 **多视角对弈/审查链** 预留 **额外模型回合**，除非用户 **随后** 显式升级该任务的复杂度（例如再次打开大脑皮层并确认进入深度编排）。
- **FR22：** 提供 **与成本/用量相关的可观测性挂点**（例如：本次任务 **内部步数/阶段计数**、或 **超过建议预算** 的开发者向警告通道）；**MVP** 不要求完整用户向计费面板，但 **不得** 完全黑盒化到无法诊断「为何烧 token」。

## Non-Functional Requirements

### Performance

- **NFR-P1：** 在典型桌面环境下，**切换大脑皮层** 须在用户可感知范围内 **即时完成**（无长时间阻塞 UI 线程；目标 **500ms 内** 完成状态切换与首帧反馈，网络与模型调用除外）。
- **NFR-P2：** 过程可视更新 **不应** 单独拖垮主聊天流式体验；事件批处理或节流策略须在实现中说明。
- **NFR-P3：** 编排默认值须 **偏向** 在 **可接受的延迟与调用次数** 下完成任务；**禁止** 将 **无上限内部多轮对话** 作为 **唯一** 完成手段（与 FR19–FR21 一致；「可接受」的具体基线在实现/测试中量化）。

### Security

- **NFR-S1：** API Key、令牌与本地配置继续遵循 **agent-diva 既有安全实践**；本增量 **不引入** 明文密钥新暴露面。
- **NFR-S2：** 黑板/内部事件等 **不可信输入** 进入模型或合成路径前须 **限长/白名单/校验**（与架构补充文档意图一致，验收以威胁模型轻量审查为准）。

### Integration

- **NFR-I1：** 新增事件与 DTO **不得破坏** 现有 LLM provider、MCP、channels 配置的 **向后兼容**（除非显式大版本）。
- **NFR-I2：** 对外集成面保持 **可列清单**（白名单事件/字段），避免「general」域无限膨胀。

### Reliability & Observability

- **NFR-R1：** **大脑皮层** 切换失败时须 **可恢复**（回滚到上一稳定状态或显式错误），**不** 静默进入未知模式。
- **NFR-R2：** 内部 trace 与用户 transcript **默认分轨**；调试内容 **不** 默认混入用户可见记录（与调研文档一致）。

### Accessibility（轻量）

- **NFR-A1：** 大脑皮层控件与神经系统入口具备 **可命名、可键盘操作**（遵循现有 GUI 组件基线；完整 WCAG 等级另立专项若需要）。

| **黑板（Blackboard）** | **共享草稿区**（常按 topic）；成员贴中间结论、工具摘要；**不等同于**对用户聊天记录。 |
| **Voice** | 内部 **审议声部**，可并行提案；用户不当作独立产品角色。 |
| **Chair** | 把多路提案 **收成一条** 对用户叙事；你们倾向 **确定性规则**（非小模型仲裁主路径）；用户自定义 Chair 规则为 **后期趣味**。 |
| **SteeringLease** | **谁有权写用户可见流**；handoff ≈ 租约转让；as_tool ≈ 租约不转、只拿结构化子结果。 |

**一手势对照：** **Handoff** = 换驾驶员；**as_tool** = 专精被调用、方向盘仍在 Person/Chair；**并行 gather** = 运行时并行探测再合并，不把并行全交给 LLM 路由。

### C. 能力与运行时（架构文档高频词）

| 概念 | 白话 |
|------|------|
| **Capability** | 成员的 **装备与契约**（工具子集、提示片段、模型档、预算/门禁）；与 **SwarmMember 正交**，**不**取消蜂群。 |
| **CapabilityRegistry** | 注册与按场景 **激活能力子集**，避免一次塞满全工具。 |
| **SwarmAction** | Shannon 式动作语义（如 `tool_call`、`publish_data`、`send_message`、`request_help`、`done`）在架构中的表达。 |
| **ConvergencePolicy** | 内层循环何时算「该停」：轮次上限、无进展、工具失败 streak、强制 done 等。 |
| **SynthesisPolicy** | Chair **如何把** 并行提案/黑板快照 **合成** 为 artifact + 对用户可见增量。 |
| **RuntimeEffect** | Swarm 决策的 **领域效果**（写用户、handoff、邮箱、黑板、调工具）；由 runtime 执行并在边界挂 Meta。 |
| **PersonOutbox** | 用户可见写的 **闸门**；应断言 lease，防泄漏。 |
| **Meta / MetaBus** | **跨轮** 事件总线（对齐 OMC hook **语义**）；**进程内 Rust**，不用 shell 当控制流；**不与 swarm 抢第二套编排权**（ADR 意图）。 |
| **双环记忆** | **Person 主记忆**（用户可见因果）+ **黑板草稿**（可丢）；压缩时优先保前者与已提交 artifact。 |
| **呈现层收束** | 借鉴 Shannon 多 station/合成，**产品面**折叠为 **单 Person** 叙事。 |
| **ToolCapability（借形）** | 能力声明字段思想（权限、危险、限流、schema…）；v0 可只取子集。 |
| **read_topics / trusted_sources** | 成员消费他成员数据时的 **可读 topic 与信任来源**，runtime 强制执行（防黑板当不可信输入）。 |
| **File-as-memory** | 长结果落盘，黑板只放 path+hash；合成时再按需读。 |

### D. 工程边界（来自架构/补充文档）

| 条目 | 含义 |
|------|------|
| **ADR-A** | `agent-diva-swarm` **不得**依赖 `agent-diva-meta`；runtime 在边界调 hook。 |
| **ADR-B** | 对用户可见归因以 **lease** 为单一真相。 |
| **ADR-C** | 黑板 v0 内存；后续可存储后端。 |
| **ADR-D** | handoff 后 **上下文继承与裁剪**（按能力/边配置）。 |
| **分层历史（tiered history）** | 缓解 token 爆炸的摘要/分层策略（借 Shannon 思路）。 |
| **用户可见 / 内部 trace / 审计 分轨** | transcript、调试、合规审计 **分存储与策略**；调试流 **不得**默认混入用户 transcript。 |

### E. 反模式与「不要抄错」（调研共识摘要）

- **不要用「能力」取代 swarm** — Capability 是装备，**协作仍是默认**。  
- **勿只抄历史 Swarm** — 需 **黑板 + 仲裁** 才能支持多声部并行审议。  
- **勿整栈抄 Shannon** — 不必默认 Temporal+Redis；协议可 **进程内降级**。  
- **勿 Crew 式无界并行且无统一 merge** — 破坏「一个人」口吻；须有 Chair/merge 规则。  
- **禁止多线程同时对用户各说各话**；**对内** 并行可以。  
- **勿绑 OMC 的 shell/脚本链** 做控制流 — 事件语义内化到 Rust。  
- **非目标示例**：用户可见十几个聊天头像；无预算/无归因/无收敛的「全员每轮发言」。  
- **路线互斥**：「仅能力、无协作」与 **内群 swarm 默认** 不兼容（调研结论）。

### F. 头脑风暴中的 v0 与范围共识（便于对齐实现）

- **先跑通**：默认合并、最简协议；租约/安全/hooks 全家桶 **跟 diva 主线**，不挡第一刀。  
- **测试从简**：假 LLM 可到「工具 JSON + 分块文本」级；属性测试/生成式状态机 **不挡 v0**。  
- **文档减负**：深潜与详尽 ADR 表 **收束为 v0 一页纸 + 产品隐喻**；其余进 Later/附录。  
- **Party 元数据**：须写清 **桌面/GUI 与神经系统**；集成面受控；**首个可演示成功时刻**待指标化（与成功标准联动）。  
- **已化解混淆**：**神经系统** = 预留全屏蜂群可视；**中控台** = 另一定位的总控，**不占**该占位。
