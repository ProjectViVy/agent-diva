# DivaGeneric UI Interaction Design

> Status: design document only. This document fixes the UI and interaction direction for DivaGeneric. It does not implement Vue, Rust, Tauri, manager routes, or storage changes.
> Related architecture: `docs/dev/genericagent/newedge/architecture.md`

## 1. Approved UI Direction

DivaGeneric keeps one primary action surface:

```text
main chat = action, planning, alignment, execution, verification
journal = archive, review, re-wakeup, navigation back to chat
```

The new `Journal` section is not a second task workspace. It is a structured archive for rhythm outputs, including daily reports, weekly reports, monthly reports, autodream summaries, and links back to the conversations and plans that produced them.

The new card-style conversation types are first-class chat interactions:

- `PlanCard`: structured plan approval and execution state.
- `OptionCard`: user decisions, learning confirmation, risk choices, and route selection.
- `ReviewCard`: rhythm/autodream review, archive confirmation, and follow-up entry.
- `JournalRefCard`: a reference card inserted into chat when a journal entry is reopened.

Plan Mode is a Diva native runtime capability. It is not a Laputa capability. Plan state belongs under `.agent-diva/plans/`. Laputa may provide personality, rhythm, constraints, and long-term continuity context, but it does not own the plan runtime.

## 2. Product Principles

### 2.1 Main Chat Is The Action Center

All active work happens in the main chat window:

- User-initiated work starts in chat.
- Diva rhythm wakeups open or resume a chat.
- Plan approval happens through a chat card.
- User alignment happens through an option card.
- Tool execution status is displayed in chat.
- Verification and acceptance happen in chat.
- Follow-up work from a journal entry returns to chat.

This keeps the user from having to decide whether a task belongs in chat, journal, plan mode, or a background page. If something requires a decision or action, Diva brings it into chat.

### 2.2 Journal Is An Archive And Re-Wakeup Surface

The Journal section stores and displays rhythm outputs:

- Daily reports.
- Weekly reports.
- Monthly reports.
- Autodream summaries.
- Links to related sessions.
- Links to related plan records.
- Evidence pointers.
- Learning candidates generated by rhythm.

Journal can reawaken an old thread, but it does not execute the work locally. Any action from Journal must return to chat and create a structured card there.

### 2.3 Cards Are Structured Conversation Messages

Cards are rendered inside `ChatView`, but they are not plain Markdown bubbles. They have stable structure, explicit status, visible evidence links, and constrained actions.

Cards must not bypass the existing architecture boundaries:

- Plan actions update `.agent-diva/plans/` through Diva runtime APIs.
- Learning actions go through Generic Core and `MemoryProvider` lifecycle boundaries.
- Rhythm actions write to `.laputa/rhythm/`.
- Mentle actions follow the existing `off/read_only/full/custom` exposure policy.

### 2.4 No Third Memory Directory

The UI must not introduce a third memory store. Storage responsibilities remain:

```text
.agent-diva/
  plans/
    <plan-id>/

.laputa/
  rhythm/
    daily/
    weekly/
    monthly/
  inbox/
```

`.agent-diva` stores Diva runtime state. `.laputa` stores personality, rhythm, long-term continuity, and learning inbox records.

## 6. Backend And Event Surface

The UI design assumes future manager routes. These routes should be introduced only when implementation starts.

Recommended journal routes:

```text
GET  /api/diary?kind=daily|weekly|monthly
GET  /api/diary/:id
POST /api/diary/:id/wakeup
```

Recommended plan routes:

```text
GET /api/plans/active
GET /api/plans/:id
```

Recommended card action route:

```text
POST /api/cards/:id/action
```

The action route should receive the card ID, action ID, and optional payload. The runtime decides whether the action updates a plan, confirms a learning candidate, starts autodream, or returns a normal chat message.

Recommended event additions:

```text
AgentEvent::CardPresented
AgentEvent::CardUpdated
AgentEvent::JournalEntryCreated
AgentEvent::PlanStateChanged
```

These events should be forwarded through the existing manager event stream rather than creating a separate UI transport.

Failure behavior:

- Card action failure must not remove the card.
- Journal loading failure must show an inline error and allow retry.
- Re-wakeup failure must return the user to chat with an error message.
- Plan state missing or corrupted must degrade to a readable error and evidence link if possible.
- Provider or memory failures must degrade and must not block the main conversation unless the requested action specifically depends on that provider.

## 7. GUI Implementation Notes

This document does not implement code, but the future implementation should map to the current GUI shape:

- Extend `NormalMode.vue` with a `journal` sidebar section.
- Add `JournalView.vue`.
- Extend `ChatView.vue` so messages may carry a `card` payload.
- Add reusable card components for plan, option, review, and journal reference cards.
- Keep normal assistant, user, tool, and system messages compatible with existing rendering.
- Keep card actions as explicit UI events, not Markdown links.
- Use existing toast/dialog surfaces for action failures and confirmations when needed.

Recommended Vue components:

The first implementation should avoid a separate right panel. If cards become too dense later, a right-side detail drawer can be added as a secondary view of the selected card, not as a new action center.

## 8. Autodream And Journal Relationship

Autodream is the process that can generate, repair, or reinterpret rhythm records. Journal is the place where those records are shown.

Direction:

```text
session/history/evidence -> autodream/rhythm -> .laputa/rhythm -> JournalView
JournalView -> re-wakeup action -> ChatView -> autodream/rhythm
```

Rules:

- Autodream may be started by rhythm schedule, user request, or journal re-wakeup.
- Autodream output must be auditable.
- Old journal entries should not be silently rewritten.
- Corrections should create follow-up records or explicit amendment records.
- Memory writes produced by autodream still go through learning candidates and user confirmation when needed.

## 9. Plan Mode And Journal Relationship

Plan Mode and Journal are connected but separate:

- Plan Mode records active task reasoning and execution under `.agent-diva/plans/`.
- Journal records rhythm summaries under `.laputa/rhythm/`.
- A journal entry can link to one or more plan IDs.
- A plan can link to journal entries that summarize its results.

Plan Mode should not write daily or monthly reports directly. It can emit evidence and completion events. Rhythm/autodream decides how those events become journal records.

## 10. Acceptance Criteria

The design is accepted when future implementers can answer the following without making new product decisions:

- Where does the user click to open Journal?
- Where does Diva show an active rhythm wakeup?
- Where does a user approve a plan?
- Where does a user confirm a learning candidate?
- Where are plans stored?
- Where are daily, weekly, and monthly reports stored?
- How does a journal entry reopen in chat?
- What card types exist?
- Which UI surface owns action execution?
- Which storage path owns plan state versus rhythm state?

Non-goals for the first UI implementation:

- No independent journal task executor.
- No full right-side workbench.
- No third memory directory.
- No direct GUI write to long-term memory files.
- No bypass around `MemoryProvider`.
- No full Mentle tool exposure in daily chat context.
