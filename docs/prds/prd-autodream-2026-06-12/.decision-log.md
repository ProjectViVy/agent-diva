## Decisions

### D1. MVP 触发范围
- **Decision**: 手动触发 + 时间门（24h），会话门放 P1
- **Rationale**: 最小可用，先验证核心价值
- **Owner**: 大湿
- **Date**: 2026-06-12

### D2. 审查模式
- **Decision**: 用户手动确认（缓存 + 决策）
- **Rationale**: 用户需要掌控记忆更新
- **Owner**: 大湿
- **Date**: 2026-06-12

### D3. AutoDream 与 mask 的关系
- **Decision**: 统一蒸馏，不隔离 mask
- **Rationale**: 先统一验证效果，再考虑隔离
- **Owner**: 大湿
- **Date**: 2026-06-12

### D4. 节律报告范围
- **Decision**: MVP 包含记忆候选 + 日报 + 周报
- **Rationale**: 完整展示节律价值
- **Owner**: 大湿
- **Date**: 2026-06-12

### D5. 失败通知
- **Decision**: 用户应收到通知
- **Rationale**: 失败不可静默
- **Owner**: 大湿
- **Date**: 2026-06-12

### D6. 与 Report System PRD 的边界（v1.1 修订）
- **Decision**: AutoDream = 数据压缩层, Report System = 用户呈现层。AutoDream 拥有日/周报**生成**, Report System 拥有月报 + 固化 + 搜索 + 展示层。
- **Rationale**: "autodream 只是一个压缩技术，而最终还是要回归到用户的可见内容中"（大湿原话）。概念正交, 两套子系统通过 markdown 文件路径契约解耦。
- **Owner**: 大湿
- **Date**: 2026-06-12
- **Reference**: `docs/architecture/scope-merge-decision.md`, Sprint Change Proposal 2026-06-12
